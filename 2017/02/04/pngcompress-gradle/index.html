      PngCompress Plugin使用

      @(tianyutingxy’s notebook)[pngquant, gradle plugin, maven]

      PngCompress 开发的目的在于，批量压缩处理项目中png图片，从一个方面所有最终apk的体积。实际使用过程中，仅处理/drawable-hdpi、/drawable-xhdpi、/drawable-xxhdpi，这几个目录的，同时设置忽略5k文件，在不包含live项目的情况下，可以多压缩出近1m的空间，压缩能力尚可，且执行操作方便。

      PngCompress 来自开源项目gradlePngCompress，该插件可对制定目录资源文件夹，进行png遍历压缩，同时提供：

      • 文件大小过滤 - 指定文件压缩过滤下限。
      • 白名单过滤 - 所有处于白名单内的文件（夹）中的内容，将予以过滤。
      • 指定额外遍历路径 - 默认情况下，该插件自动对drawable和mipmap资源包进行png遍历压缩，可额外添加其他路径。

      在接入我们的项目后，对其中的过滤功能做了相应的修改，屏蔽了对.9 patch file的压缩。PngCompress内部使用pngquant，对png进行有损压缩，但是不会破坏.9 patch file的格式，所以.9 patch file如有需要也可以进行压缩，具体原理参见github的issue。

      PngCompress原理&使用方式

      PngCompress以gradle plugin的方式，在主app中插入一个pngcompresss task ，该task可以单独执行，执行完毕后，经过压缩的png图片会覆盖原有图片。

      PngCompress Plug的项目结构如下：

      相关引用已经放到了我们私有的maven中，如果对如何使用groove编写gradle plugin有兴趣，可看看这篇博客，这里不展开了。

      pngCompress底层使用pngquant进行png的压缩，pngquant是一个有损压缩的库，虽然是有损压缩，但是其对png的质量做了较好的保留，通知最大限度的压缩png的空间，如果对市面所有png压缩方案有兴趣的话，可以看看qq音乐技术团队的一篇关于png图片压缩对比分析的博客。

      我摘出其中一段话：

      根据资料显示，tinypng、pngquant、ImageAlpha、pngnq都是有损压缩，基本采用的都是quantization算法，将24位的PNG图片转换为8位的PNG图片，减少图片的颜色数；pngcrush、optipng、pngout、advpng都是无损压缩，采用的都是基于LZ/Huffman的DEFLATE算法，减少图片IDAT chunk区域的数据。一般有损压缩的压缩率会大大高于无损压缩。

      有他们对pngquant做背书，我们就放心用吧。

      其他

      由于图片压缩基本上是一次性的活儿，平常开发与打包过程中，请勿将此功能打开，默认app中的相关配置是被注释的，如需要进行压缩，可按如上操作，进行单独的task执行，执行完毕后，请还原配置代码，并保证处于注释状态下。
